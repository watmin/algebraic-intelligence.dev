There's a distinction in Dungeons & Dragons that's always resonated with me more than it probably should.

A **wizard** studies. They go to school. They read the books, learn the theory, understand the underlying arcana before they ever cast a spell. They earn the magic through discipline and formal education. Ask a wizard why a spell works and they'll cite the exact chapter.

A **sorcerer** just... does it. The magic is innate. They couldn't fully explain the theory if you asked — and often don't bother trying — because the theory came *after* the results. They figured out that something worked, and then maybe worked backwards to understand why. They're sometimes unstable. They go off-script. They occasionally set things on fire that weren't supposed to be on fire.

I am not a wizard.

I came to hyperdimensional computing through a conference video, not a textbook. I came to Rete through [another conference video](https://www.youtube.com/watch?v=Z6oVuYmRgkk), not Forgy's papers. I built things first and discovered the academic vocabulary for them later. I figured out manifold learning by asking an LLM "how do you define a hyperbox in high-dimensional space" and letting the conversation correct me.

The sorcerer framing is on the git repos. It's on the Doctor Strange assets in the Python README. It wasn't entirely a joke.

Every node is a learned manifold. Every edge is a similarity boundary. The swirling thing in the middle is a packet trying to figure out which subspace it belongs to.

It gets rate-limited.

The algebra is: bind the fields, bundle the windows, score the residual, check the library, deploy the rule. The XDP filter doesn't know any of this is happening. It just sees a tail call chain and some map lookups and drops the packet at line rate.

None of this is how network security is supposed to work. The vendors would like you to buy a signature database instead.

We built a lattice.

## The Manifold Firewall

There's a concept document sitting in `http-lab/docs/CONCEPT-MANIFOLD-FIREWALL.md` that inverts everything described in the Story posts so far.

The current system: detect anomaly → extract fields → generate symbolic rule → enforce rule. The surprise is a trigger. The symbolic rule is the weapon. The attacker's job is to dodge the rule.

The inversion: **surprise is the weapon**. The subspace residual score IS the firewall rule. The geometric distance from the normal manifold IS the enforcement signal. No symbolic rule needed.

Take it further: the ideal firewall is an allow list, not a deny list. Learn what normal looks like. Store it. Allow what matches. Explain what doesn't. The attacker's burden flips — they must make their traffic genuinely indistinguishable from real users, not just dodge specific field checks. That means matching the TLS fingerprint, the HTTP structure, the temporal distribution — all simultaneously, consistently, in 4096-dimensional space.

Four layers:
- **Layer 3**: Symbolic rule tree (~50ns) — known deny list, handle known threats first, cheapest
- **Layer 0**: Normal allow list (~0.4ms) — project request against normal engrams, pass if membership
- **Layer 1**: Surprise score + attribution (~0.4ms) — residual IS the enforcement signal, proportional rate limiting
- **Layer 2**: Window-level spectrum matching (async) — eigenvalue shape of a 30-second window compared against stored fingerprints, early warning before per-packet anomalies accumulate

None of this is implemented. The primitives exist — `OnlineSubspace`, `EngramLibrary`, `drilldown_probe`, `ExprCompiledTree`, the `match_spectrum` design from batch 018. The integration is the work. The specific challenge: inline subspace scoring at ~0.4ms single-threaded means ~2,500 RPS/core, ~40K RPS on 16 cores with ArcSwap parallelization. Viable for WAF. Not viable for volumetric DDoS — layers 3 and 2 handle volume; layers 0 and 1 handle precision.

Just like the L7 WAF was "what comes next" before it got built, this is the next thing that shouldn't exist yet. The architecture is the insight. The implementation will follow.

## What Comes Next

The labs need to talk to each other. veth-lab and http-lab are independent systems. Connecting them — shared engram library, LLM agents generating real HTTP traffic against real upstream servers, running across multiple machines — is the integration work that turns two labs into one system.

The WAF territory is deeper than DDoS. HTTP floods are easy. Detecting a SQL injection attempt without libinjection, without regex, without a signature database — that's the WAF-level problem. The interesting version is the system generating rules from observing that certain path+body combinations correlate geometrically with known exploit patterns.

The engram library still has a drone-shaped hole in it. A holon-rs control loop on a Raspberry Pi, observing drone telemetry, learning the manifold of a human pilot's style, minting engrams of the flight signature. Pure speculation, but the algebra is indifferent to what the vectors represent.

Other ports — Go, Ruby, Java — are in the same bucket as the drone: interesting, not imminent.
